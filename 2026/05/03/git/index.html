注意事项：

注意事项 说明
远程仓库要为空 不要在 GitHub 上勾选 “Add README” 等初始化文件
分支名要一致 检查本地分支名：git branch，默认可能是 master 或 main
地址要正确 用 git remote -v 检查远程地址
首次要用 -u 建立本地分支和远程分支的追踪关系，以后只需 git push

注意事项：

注意事项 说明
推送前先拉取 养成 git pull 习惯，避免冲突
commit 信息要清晰 如 “修复登录bug”、”添加用户模块”
不要提交大文件 视频、压缩包等用 .gitignore 忽略
敏感信息不要提交 API密钥、密码等用环境变量
定期推送 完成一个功能点就推送，不要积累太多

三、冲突解决方法

冲突是怎么产生的？

解决步骤：

1. 拉取时发现冲突

2. 查看冲突文件

3. 打开冲突文件，找到冲突标记

<<<<<<< HEAD 到 ======= 之间：是你本地当前分支（HEAD）的内容 → 你的内容：Hello Git

======= 到 >>>>>>> origin/master 之间：是远程分支（origin/master）的内容 → 远程的内容：Hello GitHub

4. 手动解决冲突

冲突原因：
Git 不知道要保留“Hello Git”还是“Hello GitHub”，所以停下来让你手动选择（或合并两者）。

你需要做的：
编辑这个文件，把冲突标记（<<<<<<<、=======、>>>>>>>）删掉，并改成你想要最终保留的样子。例如：

只保留你的：Hello Git

只保留远程的：Hello GitHub

两者都保留：Hello Git Hello GitHub

改完后保存文件，然后执行 git add <文件名> 和 git commit 完成合并。

5. 标记为已解决

6. 完成合并提交

7. 推送

避免冲突的技巧：

技巧 做法
频繁拉取 每天开始工作前 git pull
分工明确 尽量不要多人改同一个文件
小步提交 完成一个小功能就推送
及时沟通 要改公共文件时先通知团队

不想解决冲突？强制推送（危险！）

只在确定要覆盖远程时才用，团队协作中禁止使用！

场景 命令

推送前拉取 git pull origin master
查看状态 git status
查看远程 git remote -v
